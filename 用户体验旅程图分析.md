# MarkPage — Chrome 书签标签页插件：用户体验旅程图分析

## 产品概述

MarkPage 是一款 Chrome 新标签页插件，核心目标是帮助用户高效管理浏览器书签，提供：
1. 自定义常用页面（美观、快捷访问）
2. 全部书签浏览与管理
3. AI 自动整理分类（用户自配 API Key）
4. 智能搜索（书签搜索 + Google 搜索）

---

## 一、用户角色画像

| 角色 | 特征 | 核心诉求 |
|------|------|----------|
| **效率型用户** | 书签 200+，日常高频使用 10-20 个网站 | 快速找到目标网站，不想在书签栏翻找 |
| **收藏型用户** | 书签 500+，习惯"先收藏再说" | 书签积累混乱，找不到之前收藏的内容 |
| **轻度用户** | 书签 < 50，偶尔使用 | 希望新标签页美观实用，快速直达常用站点 |

## 二、用户体验旅程图

### 阶段 1：发现与安装

```
用户行为：  搜索插件 → 查看介绍 → 安装插件 → 首次打开新标签页
情绪曲线：  😐 → 🤔 → 😊 → ❓
触点：      Chrome Web Store / 推荐文章
```

**痛点：**
- P1: 用户对"替换新标签页"有顾虑 — 担心影响原有浏览体验
- P2: 安装后首次打开，面对空白页面不知从何开始
- P3: 不清楚插件需要哪些权限，对"读取书签"权限有安全顾虑

**核心需求：**
- ✅ 清晰的功能预览和价值主张
- ✅ 首次启动引导（Onboarding Flow）
- ✅ 权限说明透明化

---

### 阶段 2：初始配置

```
用户行为：  设置常用网站 → 配置 AI API Key → 选择分类偏好 → 导入现有书签
情绪曲线：  🤔 → 😣 → 😤 → 😊
触点：      设置页面 / 引导流程
```

**痛点：**
- **P4（严重）: API Key 配置门槛高** — 普通用户不知道什么是 API Key，不知道去哪里获取，不知道选哪个模型、如何付费。这是最大的用户流失点
- P5: 手动设置常用网站繁琐 — 需要逐个添加 URL、图标、名称
- P6: 现有书签数量庞大时，首次整理耗时长，用户可能放弃

**核心需求：**
- ✅ AI 配置需要极简化的引导教程（分步骤、带截图）
- ✅ 支持多个 AI 服务商（OpenAI / Claude / DeepSeek / 国内模型），降低选择成本
- ✅ 智能推荐常用网站（基于浏览历史 / 已有书签自动识别高频站点）
- ✅ 书签批量导入 + 后台静默整理（不阻塞用户使用）

---

### 阶段 3：日常使用 — 快速访问

```
用户行为：  打开新标签 → 扫视常用网站 → 点击目标网站
情绪曲线：  😊 → 😊 → 😊（理想状态）
           😊 → 😤 → 😡（找不到时）
触点：      新标签页主界面
```

**痛点：**
- P7: 常用网站超过一屏时，视觉混乱，找不到目标
- P8: 网站图标不统一（有的有 favicon，有的没有），视觉体验差
- P9: 无法快速区分"工作"和"个人"场景的常用站点
- P10: 页面加载速度 — 新标签页必须毫秒级响应，任何延迟都是致命的

**核心需求：**
- ✅ 支持分组 / 文件夹方式组织常用网站（工作、学习、娱乐等）
- ✅ 统一的图标展示方案（自动获取高清 favicon + 首字母兜底）
- ✅ 页面性能极致优化 — 目标：< 100ms 首屏渲染
- ✅ 支持拖拽排序、自定义布局
- ✅ 视觉设计精致，支持主题/壁纸自定义

---

### 阶段 4：日常使用 — 搜索

```
用户行为：  输入关键词 → 查看搜索建议 → 选择书签结果 or 跳转 Google
情绪曲线：  🤔 → 😊（找到了）/ 😤（找不到）
触点：      搜索框 + 搜索结果面板
```

**痛点：**
- **P11（严重）: 搜索结果混合展示时，用户分不清哪个是书签、哪个是搜索建议**
- P12: 书签搜索只匹配标题，不匹配 URL 或标签，召回率低
- P13: 搜索框需要额外点击才能聚焦 — 应该打开新标签页就能直接输入
- P14: 模糊搜索能力弱，拼错关键词就找不到

**核心需求：**
- ✅ 搜索结果分区展示：书签结果 + Google 搜索 明确分开
- ✅ 全文搜索：匹配标题、URL、标签、描述
- ✅ 打开新标签页自动聚焦搜索框（可配置）或支持快捷键快速聚焦
- ✅ 模糊匹配 + 拼音搜索（中文用户场景）
- ✅ 搜索历史 / 最近访问快速推荐

---

### 阶段 5：日常使用 — 新增书签

```
用户行为：  收藏网页 → 弹出分类建议 → 确认/修改分类 → 保存
情绪曲线：  😊 → 🤩（分类准确）/ 😤（分类错误需手动改）
触点：      Chrome 收藏按钮 / 插件弹窗
```

**痛点：**
- **P15（严重）: Chrome 原生收藏只能放到最近使用的文件夹，分类全靠手动**
- P16: AI 分类建议如果不准确，反复修改比手动分类更烦
- P17: 收藏时需要等 AI 返回结果，网络延迟影响体验
- P18: 用户的分类体系因人而异，AI 预设分类可能与用户习惯不匹配

**核心需求：**
- ✅ 收藏时自动弹出智能分类建议（基于用户已有分类体系学习）
- ✅ AI 分类需要从用户的历史行为中学习，而非通用分类
- ✅ 异步处理：先保存后分类，不阻塞用户操作
- ✅ 支持一键确认 + 快速修改分类
- ✅ 分类建议需展示置信度，让用户知道 AI 有多确定

---

### 阶段 6：日常使用 — 书签管理

```
用户行为：  浏览全部书签 → 查找/整理/删除 → 批量操作
情绪曲线：  🤔 → 😣（书签太多太乱）→ 😊（整理完毕）
触点：      书签管理页面
```

**痛点：**
- **P19（严重）: 书签数量庞大时，树形结构展开很深，无法快速定位**
- P20: 存在大量死链（404 页面），浪费认知资源
- P21: 重复收藏的网站没有提醒
- P22: 批量操作困难 — Chrome 原生书签管理器几乎没有批量功能

**核心需求：**
- ✅ 多种视图：列表视图、卡片视图、树形视图 自由切换
- ✅ 死链检测：自动标记/清理失效书签
- ✅ 重复检测：收藏时提醒已存在
- ✅ 批量操作：批量移动、删除、打标签
- ✅ AI 一键整理：对全部书签重新分类（可预览、确认后执行）

---

### 阶段 7：持续使用 — 维护与优化

```
用户行为：  定期整理 → 调整分类 → 更新常用 → 更换 AI 模型
情绪曲线：  😐 → 😊（越来越好用）/ 😤（越来越乱）
触点：      设置页面 / 书签管理
```

**痛点：**
- P23: 分类体系随时间变化，旧的分类不再适用，需要重构
- P24: API Key 额度用完后无提示，AI 功能静默失效
- P25: 数据只存在 Chrome 中，换电脑/重装浏览器后丢失

**核心需求：**
- ✅ AI 分类体系可随时调整，支持合并/拆分/重命名分类
- ✅ API 用量监控 + 余额不足提醒
- ✅ 书签数据导出/导入（JSON/HTML 格式）
- ✅ 支持 Chrome Sync 同步（或自建同步方案）

## 四、核心设计原则（基于痛点推导）

1. **速度至上** — 新标签页是高频入口，性能是底线，不是特性
2. **渐进式 AI** — AI 功能不能是门槛，应该是惊喜。无 API Key 时也能正常使用，配置后体验升级
3. **搜索即入口** — 搜索框是第一交互点，输入即见结果，零等待
4. **分类自适应** — 不强推通用分类，从用户行为中学习个性化分类体系
5. **操作不阻塞** — 所有 AI 操作异步化，先完成用户动作，后台处理智能逻辑

### 竞品的共同缺陷（= MarkPage 的机会）

1. **无 AI 智能分类** — 所有主流竞品都依赖手动分类，这是最大的差异化空间
2. **数据可靠性差** — OneTab/Workona 均有数据丢失投诉，信任是关键
3. **付费墙引发用户不满** — Momentum、Speed Dial 2、Toby 都因激进变现被差评
4. **新标签页 + 书签管理割裂** — 市面上没有产品同时做好"快捷访问"和"深度书签管理"

### AI 书签类新兴竞品

Bookmark Genie、BookmarkBuddy、Autolicious 等已尝试 AI 分类方向，但均处于早期阶段，评价量少，功能粗糙。**市场窗口仍然开放**。

### Prompt 设计策略

**核心思路：不是让 AI 从零分类，而是让 AI 在用户已有体系中选择最合适的位置**

```
Prompt 结构：
1. 系统角色：你是一个书签分类助手
2. 用户的分类树：[从 Chrome 书签文件夹结构提取]
3. 历史分类样本：[最近 20 条用户确认的分类]
4. 待分类网页信息：
   - 标题: xxx
   - URL: xxx
   - 描述(meta description): xxx
5. 输出格式：JSON { category, confidence, alternatives }
```

**关键设计决策：**

| 决策点 | 方案 | 理由 |
|--------|------|------|
| 输入信息 | 标题 + URL + meta description | 不需要抓取全文，节省 Token 成本，速度快 |
| 分类基准 | 用户已有文件夹结构 | 尊重用户习惯，避免"AI 觉得你应该这么分" |
| 学习机制 | 记录用户修改，作为 few-shot 样本 | 越用越准，无需微调模型 |
| 新分类 | AI 可建议创建新分类（需用户确认） | 覆盖用户体系中的空白 |
| 失败兜底 | 放入"未分类"文件夹，后台重试 | 永远不阻塞用户操作 |

### 支持的 AI 服务商

| 服务商 | 模型 | 特点 | 成本估算（每次分类） |
|--------|------|------|---------------------|
| OpenAI | gpt-4o-mini | 性价比最高，速度快 | ~$0.0001 |
| Anthropic | claude-haiku | 中文理解好 | ~$0.0001 |
| DeepSeek | deepseek-chat | 国内可用，中文优秀 | ~$0.00005 |
| 兼容 OpenAI 格式 | 自定义 | 覆盖更多服务商 | — |

**推荐默认：支持 OpenAI 兼容格式 → 一套代码覆盖 90% 的 LLM 服务商**

- ESC 关闭搜索浮层，回到首页
- 搜索结果支持键盘上下导航 + 回车打开

- **操作可撤销**：书签移动后显示 "撤销" Toast，30 秒内可回退
